# 06 — Website Distribution

## Goal

The canonical consumer-facing artifact is `Essenfont-Regular.otc`,
distributed from `https://essenfont.github.io` (the
`essenfont/essenfont.github.io` repo). The website offers:

1. **One-click download** of the OTC file (~40-60 MB).
2. **Per-plane TTF downloads** for clients that can't consume OTC
   (older Linux fontconfig, some font managers).
3. **Live coverage map** updated from the build's coverage.json.
4. **Donor registry** with provenance per subfont.

## File layout

The website (a Vue 3 + Vite app) currently serves `Essenfont-Regular.ttf`
from `dist/` via ViteSSG. The new structure:

```
essenfont.github.io/
├── public/
│   ├── releases/
│   │   └── v0.2/
│   │       ├── Essenfont-Regular.otc            # canonical
│   │       ├── Essenfont-BMP.ttf                # per-plane fallbacks
│   │       ├── Essenfont-SMP.ttf
│   │       ├── Essenfont-SIP.ttf
│   │       ├── Essenfont-TIP.ttf
│   │       ├── Essenfont-SSP.ttf
│   │       └── coverage.json                    # per-build manifest
│   └── coverage.json -> releases/v0.2/coverage.json (symlink)
├── src/
│   ├── components/
│   │   ├── DownloadCard.vue                     # canonical download
│   │   ├── CoverageMap.vue                      # existing, refresh data
│   │   └── SubfontList.vue                      # new — lists subfonts
│   └── pages/
│       └── Index.vue
└── scripts/
    └── fetch-release.js                         # CI hook — copies artifacts
```

The `.otc` is **not committed to git** — it's a binary release
artifact fetched by the `fetch-release.js` script during the website
CI build. Same model as today's TTF flow.

## CI flow

The build + publish pipeline runs in two repos:

### essenfont/essenfont (build repo)

1. Tag pushed (`v0.2`).
2. CI runs `scripts/build.rb` → `Essenfont-Regular.otc` + per-plane TTFs.
3. CI runs `scripts/emit_coverage_manifest.rb` → `coverage.json`.
4. CI uploads all artifacts to the GitHub Release.
5. CI triggers downstream dispatch via `repository_dispatch`.

### essenfont/essenfont.github.io (website repo)

1. Receives `repository_dispatch` with the release payload.
2. Runs `scripts/fetch-release.js`, which downloads:
   - `Essenfont-Regular.otc` → `public/releases/v0.2/`
   - Per-plane TTFs → same dir
   - `coverage.json` → same dir
3. Updates `public/coverage.json` symlink to the new release.
4. Commits to `main` (the website repo's `main`, not essenfont's).
5. Deploys via GitHub Pages.

The website's `main` is the deploy branch. Binary artifacts under
`public/releases/` are gitignored from the website repo (size
management) — fetched fresh on each build. Or committed via Git LFS
if size grows past Pages limits.

## @font-face usage

For web authors who want to embed essenfont directly (no install):

```css
@font-face {
  font-family: 'essenfont';
  src: url('https://essenfont.github.io/releases/v0.2/Essenfont-BMP.woff2') format('woff2');
  font-display: swap;
  unicode-range: U+0000-FFFF;
}
@font-face {
  font-family: 'essenfont';
  src: url('https://essenfont.github.io/releases/v0.2/Essenfont-SMP.woff2') format('woff2');
  font-display: swap;
  unicode-range: U+10000-1FFFF;
}
/* ... per-plane TTFs, each WOFF2-encoded ... */

body {
  font-family: -apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif, 'essenfont';
}
```

The OTC is **not** web-embeddable — `@font-face` does not accept
collection formats. Web users consume per-plane WOFF2s (TTFs
converted to WOFF2 in CI). This is documented in the website's
"Web usage" section.

## Coverage map refresh

The existing `CoverageMap.vue` reads `coverage.json` and renders
per-block coverage. The new `coverage.json` adds per-subfont
breakdown. The map can now color-code:

- **Green** — covered in the OTC (always; if any subfont covers it).
- **Teal** — partially covered (e.g., donor cmap covers 60% of
  block; the rest are .notdef backfill).
- **Gray** — uncovered.

Tooltip now shows: "Block X (U+AAAA..U+ABCD): covered by subfont
BMP via donor noto_sans_arabic, 100% of block assigned (250/250)."

## Release checklist (per release)

For each `vX.Y` tag:

- [ ] Run build, confirm no `validate_and_repair_cmap` warnings.
- [ ] `fc-query Essenfont-Regular.otc` reports ≥5 faces.
- [ ] Per-face `maxp.num_glyphs ≤ 65,535`.
- [ ] Union of face cmaps = input cp_map.
- [ ] `coverage.json` emitted with correct counts.
- [ ] Website CI runs `fetch-release.js`, deploys.
- [ ] Manual smoke test: open website, click "Download OTC", verify
  file size and MD5 match CI artifact.
- [ ] Manual smoke test: install on macOS, verify font shows up as
  5 faces in Font Book.
- [ ] Update `CHANGELOG.md` with coverage delta and donor changes.

## Backwards compatibility with v0.1 URLs

Existing v0.1 download links (`/releases/v0.1/Essenfont-Regular.ttf`)
keep working — the v0.1 TTF stays in the website's `public/releases/`
in perpetuity (or until explicitly retired). Users on v0.1 don't
break; they just don't get the coverage improvements of v0.2.

New users are directed to v0.2 OTC by default. The DownloadCard
"Looking for a single TTF?" links default to v0.2 per-plane TTFs.
